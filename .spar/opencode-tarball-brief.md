# OpenCode Setup Tarball — Decision Brief

## Feature

`generate-tarball.sh` in the home-manager repo root that produces a self-contained
`opencode-setup.tar.gz` for colleagues who want the full OpenCode CLI setup without
Nix or home-manager. Extract to `~/` and everything is in place — no further steps.

## Context

Not all colleagues use Nix/home-manager. They need a way to get the full global
OpenCode CLI setup (agents, skills, commands, tools, config) without symlinks or
Nix. The home-manager repo is the single source of truth for all global OpenCode
configuration. Colleagues can keep up with upstream by pulling both repos and
re-running the script on their own machine.

## Key Decisions

- Script lives in `home-manager/` repo root as `generate-tarball.sh`
- No symlinks anywhere — all files are copied into the tarball as regular files
- `node_modules` is bundled via a fresh `bun install` run by the script against
  `home-manager/opencode/package.json` inside the staging directory; this makes
  the tarball fully self-contained and platform-correct for the machine that
  generated it
- Script checks for `../ai-coding` (sibling directory); if absent it clones
  `https://github.com/vansweej/ai-coding.git` there automatically
- Hardcoded relative path `../ai-coding` for sourcing `pipeline.ts` and
  `opencode.json` — simplest setup for now, can be made configurable later
- Tarball root path is `.config/opencode/` so `tar xzf opencode-setup.tar.gz -C ~/`
  places everything correctly under `~/.config/opencode/`
- Tarball is the deliverable — colleagues inspect contents first (`tar tf
  opencode-setup.tar.gz` or extract to a temp dir) before applying
- `README-install.md` is generated inside the tarball by the script (not manually
  maintained), so it always reflects the current state of the setup
- Each person runs the script on their own machine and platform — no cross-platform
  tarball; `node_modules` may contain platform-specific native binaries
- OpenCode CLI only — VS Code support is a separate future effort
- Bun is a prerequisite; colleagues must install it via their own package manager
  (brew, apt, etc.) before running the script

## Generated README-install.md Content (outline)

The script generates a `README-install.md` inside the tarball covering:
1. Prerequisites — OpenCode CLI installed, Bun installed
2. Inspect the tarball before applying — `tar tf opencode-setup.tar.gz`
3. Back up existing config — `cp -r ~/.config/opencode ~/.config/opencode.bak`
4. Extract — `tar xzf opencode-setup.tar.gz -C ~/`
5. Set `AI_CODING_MONOREPO` env var in shell profile (needed for pipeline tool)
6. Add `$HOME/.opencode/bin` to PATH in shell profile
7. Restart OpenCode after extraction

## Open Questions

- Should the tarball filename include a date or git-hash stamp (e.g.
  `opencode-setup-2026-04-29.tar.gz`) for versioning?
- Should the script fail fast with a clear error if `bun` is not installed?
- When colleagues re-run and re-extract to update, stale renamed/removed files
  remain in `~/.config/opencode/`. Document `rm -rf ~/.config/opencode` before
  re-extracting, or add a `--clean` flag that does this automatically?

## Rejected Alternatives

| Alternative | Reason rejected |
|---|---|
| Direct install script | Colleagues want to review before applying — tarball gives that inspection step |
| Symlink-based setup | Rejected by user; must work without repos present after extraction |
| Cross-platform single tarball | `node_modules` contains platform-specific native binaries |
| Docs-only approach | Too error-prone for 30+ files to copy manually |
| Makefile / just recipe | No meaningful benefit over a standalone shell script for this use case |

## Risks

1. **`ai-coding` repo drift** — if `pipeline.ts` or its API changes, colleagues
   must re-run the script to pick up the update; there is no automatic
   notification mechanism. Mitigate: document in `home-manager/README.md` that
   the script should be re-run after pulling either repo.

2. **Bun version mismatch** — `@opencode-ai/plugin` may require a minimum Bun
   version. The script does not currently validate this. Mitigate: add a version
   check and clear error message to the script.

3. **Stale files on update** — if a skill or agent is renamed or removed
   upstream, re-extracting the tarball will not delete the old file from
   `~/.config/opencode/`. Mitigate: document a clean re-install procedure in
   `README-install.md`.

4. **`node_modules` size** — bundling `node_modules` increases tarball size.
   Acceptable trade-off for zero-dependency extraction; no mitigation needed.

## Recommended Next Steps

1. Implement `generate-tarball.sh` in `home-manager/` repo root:
   - Check for `bun` on PATH; fail with a clear message if absent
   - Check for / clone `../ai-coding`
   - Create a staging directory mirroring `.config/opencode/`
   - Copy all files from `home-manager/opencode/` into staging
   - Copy `pipeline.ts` from `../ai-coding/.opencode/tools/`
   - Copy `opencode.json` from `../ai-coding/opencode/mappings/`
   - Run `bun install` in the staging directory to produce `node_modules`
   - Generate `README-install.md` into staging
   - Pack staging into `opencode-setup.tar.gz` with `.config/opencode/` as root
   - Remove staging directory

2. Document the colleague workflow in `home-manager/README.md`:
   - Install prerequisites (OpenCode CLI, Bun)
   - Clone `home-manager` repo
   - Run `./generate-tarball.sh` (auto-clones `ai-coding` if needed)
   - Inspect tarball contents
   - Extract and configure shell profile

3. Resolve the three open questions above before or during implementation.
